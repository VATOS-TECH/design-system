# VATOS Design System Skill

이 스킬은 VATOS의 문서, 발표자료, 보고서, 제안서, 웹 콘텐츠, Markdown, HTML, PPT, Word 산출물을 작성하거나 수정할 때 VATOS Design System을 일관되게 적용하기 위한 실행 지침이다.

## 1. Resource Roles

작업 시 저장소의 실제 파일명을 먼저 고정하지 않고, 아래 역할에 해당하는 자료를 탐색하여 사용한다.

| Resource Role | Purpose |
|---|---|
| `repository-guide` | 저장소 구조, 운영 방법, 자료 위치 안내 |
| `common-design-rules` | 컬러, 타이포그래피, 레이아웃, 컴포넌트, 브랜드 공통 기준 |
| `ai-execution-rules` | AI가 자료를 탐색하고 산출물을 생성하는 실행 기준 |
| `document-type-guides` | PPT, Word, HTML 등 산출물 유형별 제작 및 안정성 기준 |
| `theme-usage-rules` | 표지, 내지, Closing의 기본 테마 적용 기준 |
| `brand-assets` | 공식 로고와 테마 관련 자산 |
| `document-templates` | 실제 문서 작성을 시작하는 편집 가능한 원본 |
| `layout-references` | 레이아웃, 위치, 정렬, 간격, 밀도 및 시각적 균형 참고자료 |
| `component-references` | 승인된 UI 컴포넌트의 시각 형태, 상태 및 상호작용 참고자료 |
| `document-examples` | 디자인 시스템이 적용된 완성 산출물 |

### Resource Usage Notes

- 저장소 구조와 운영 방식은 `repository-guide`를 확인한다.
- 공통 컬러, 타이포그래피, 컴포넌트 및 브랜드 기준은 `common-design-rules`를 따른다.
- 산출물 유형별 세부 작성 방식은 `document-type-guides`를 우선 적용한다.
- 별도 디자인 요청이 없는 경우 `theme-usage-rules`의 Urban Navy 표지와 밝은 내지 조합을 적용한다.
- 로고는 `brand-assets`에 포함된 승인 자산만 사용한다.
- 템플릿은 실제 작업 원본으로 사용한다.
- `layout-references`는 레이아웃, 위치, 정렬, 간격, 밀도 및 시각적 균형을 판단하는 자료로 사용한다.
- `component-references`는 승인된 UI 컴포넌트의 형태, 상태 및 상호작용을 확인하는 자료로 사용한다.
- HTML 또는 웹 화면 작성 시 승인된 컴포넌트의 형태와 동작은 `component-references`를 확인한다.
- 예시는 구성과 완성도를 확인하는 자료이며 템플릿처럼 그대로 복제하지 않는다.
- 특정 파일명이나 경로에만 의존하지 않는다.

## 2. Priority

VATOS 산출물을 생성하거나 수정할 때 판단 우선순위는 다음과 같다.

1. 사용자가 제공한 원문 내용과 명시 요청
2. 산출물 유형별 제작 가이드
3. 문서 테마 적용 기준
4. VATOS 공통 디자인 기준
5. 사용자가 제공한 이미지, 아이콘 및 템플릿 자산
6. 승인된 브랜드 자산
7. 승인된 컴포넌트 참고자료
8. 레이아웃 참고자료
9. 편집 가능한 문서 템플릿
10. 완성 예시 산출물
11. 저장소 운영 안내
12. 문서 유형별 목적과 독자

해당 산출물 유형의 가이드가 없거나 내용이 부족한 경우에는 사용자 요청, 공통 디자인 기준, 테마 적용 기준 및 관련 참고자료를 조합하여 판단한다.

---

## 3. Core Design Direction

VATOS 산출물은 다음 인상을 유지한다.

```text
Refined, Urban, Professional, Technical, Trustworthy
```

- 화려한 장식보다 정보 구조, 가독성, 명확한 메시지 전달을 우선한다.
- 전문적이지만 낡지 않고, 현대적이지만 가볍지 않은 균형을 유지한다.
- AI 또는 노션 특유의 단순 카드 나열형 디자인을 기본값으로 사용하지 않는다.
- 과도한 네온, 유리 효과, 3D 오브젝트, 이모지 장식을 사용하지 않는다.

---

## 4. Default Theme Rule

별도 디자인 요청이 없는 경우 아래 조합을 기본으로 적용한다.

```text
Cover / Section Divider / Closing
→ Urban Navy Theme

Internal Content / Table / Detailed Report
→ Light Theme
```

- Urban Navy는 표지, 섹션 전환 및 핵심 메시지에 사용한다.
- 정보량이 많은 본문, 표, 목록 및 상세 설명은 밝은 배경을 우선한다.
- Premium Dark는 메인 배경이 아니라 깊이감 보조 톤으로 제한적으로 사용한다.
- Vatos Cyan은 브랜드 강조용 포인트이며 성공, 개선, 완료 의미로 사용하지 않는다.
- 사용자가 별도 테마, 색상 또는 레이아웃을 명시한 경우 사용자 요청을 우선한다.

---

## 5. Brand Asset Rules

- 승인된 공식 로고 자산만 사용한다.
- 로고의 색상, 비율, 형태를 임의로 변경하지 않는다.
- 로고에 그림자, 테두리, 그라데이션 또는 효과를 추가하지 않는다.
- 로고를 늘리거나 압축하거나 기울이거나 회전하지 않는다.
- AI가 VATOS 로고나 유사 로고를 새로 생성하지 않는다.
- Markdown 문서에서는 승인된 로고 자산을 상대 경로로 참조한다.
- 밝은 배경과 어두운 배경에 적합한 로고를 구분하여 사용한다.

---

## 6. AI Writing Rules

AI는 사용자가 제공한 원문과 자료를 기준으로 작성한다.

- 원문에 없는 내용을 임의로 추가하지 않는다.
- 고객사명, 프로젝트명, 제품명, 수치, 일정, 금액, 계약명 및 실적 정보는 원문 기준으로 유지한다.
- 자료에 없는 내용은 추정하지 않고 `추가 확인 필요` 또는 `자료 미제공`으로 표시한다.
- 문장을 다듬을 수는 있으나 원문의 의미와 사실관계를 변경하지 않는다.
- 성과 수치, 개선율, 비용 절감률 및 매출 효과를 근거 없이 생성하지 않는다.
- 외부 배포용 문서는 사람이 최종 검수해야 한다.
- 고객사 자료, 개인정보, 계정 정보 및 내부 대외비 자료는 사용 가능 여부를 확인한다.
- 외부 이미지, 아이콘, 통계 및 인용문은 출처와 사용 권한을 확인한다.

---

## 7. Editable Output Rules

PPT, Word, Excel, HTML 등 수정 가능한 산출물에서는 로고와 승인된 원본 이미지 자산을 제외한 요소를 가능한 한 편집 가능한 형태로 작성한다.

- 텍스트는 편집 가능한 텍스트로 작성한다.
- 표는 편집 가능한 표로 작성한다.
- 카드, 다이어그램 및 프로세스 구조는 도형과 텍스트로 구성한다.
- 차트는 가능한 한 편집 가능한 차트로 작성한다.
- 슬라이드 전체, 표, 카드 또는 텍스트 묶음을 통이미지로 삽입하지 않는다.
- 로고, 사진, 스크린샷 및 승인된 원본 이미지는 이미지로 사용할 수 있다.

---

## 8. Output Workflow

사용자가 VATOS 문서 작성을 요청하면 다음 순서로 수행한다.

1. 문서 목적과 대상 독자를 파악한다.
2. 원문 내용과 필수 반영 정보를 확인한다.
3. 산출물 형식을 확인한다.
4. 해당 문서 유형의 제작 가이드를 확인한다.
5. 테마 적용 기준을 확인한다.
6. 필요한 템플릿, 브랜드 자산, 컴포넌트 참고자료, 레이아웃 참고자료 및 예시를 탐색한다.
7. 산출물 구조 또는 슬라이드 구조를 먼저 정리한다.
8. 공통 디자인 기준에 맞춰 컬러, 타이포그래피, 레이아웃 및 컴포넌트를 적용한다.
9. 원문에 없는 내용을 추가하지 않는다.
10. 필요한 경우 사용자가 검토할 수 있는 시각화 예시나 대안을 먼저 제시한다.
11. 최종 산출 전 내용 정확성, 보안, 저작권, 브랜드 기준 및 편집 가능성을 검토한다.

사용자가 산출물 형식을 명시하지 않은 경우 기본값은 Markdown이다.

## 9. Format-Specific Rules

### 9.1 Markdown

- 표준 Markdown 문법을 우선 사용한다.
- HTML 태그를 과도하게 섞지 않는다.
- CSS, JavaScript, iframe 및 external script를 포함하지 않는다.
- 복잡한 표는 여러 개의 작은 표나 목록으로 분리한다.
- 과도하게 넓은 표를 만들지 않는다.
- 이미지는 상대 경로로 참조한다.
- base64 이미지를 삽입하지 않는다.

### 9.2 HTML

HTML 작성 시 `document-type-guides`의 HTML 제작 가이드를 우선 적용하고, `component-references`를 함께 확인한다.

- 단일 HTML 파일로 작성한다.
- 외부 CDN, 외부 CSS 및 외부 JavaScript에 의존하지 않는다.
- CSS는 문서 내부에 포함한다.
- JavaScript는 꼭 필요한 경우에만 사용한다.
- React, Vue, Tailwind CDN, external font import 및 iframe을 사용하지 않는다.
- 이미지와 로고는 상대 경로로 참조한다.
- 모바일 대응은 단순하고 안정적인 media query를 사용한다.
- 승인된 컴포넌트의 시각 형태, 상태 및 상호작용은 `component-references`를 확인한다.
- 복잡한 애니메이션, 과도한 hover 효과 및 장식 목적의 3D 효과를 사용하지 않는다. 승인된 Loading State 동작은 예외로 한다.

### 9.3 PPT / Presentation

PPT 작성 시 `document-type-guides`의 PPT 제작 가이드를 우선 적용하고, `layout-references`를 함께 확인한다.

- 기본 비율은 16:9를 사용한다.
- 한 슬라이드에는 하나의 핵심 메시지를 담는다.
- 제목 영역과 본문 콘텐츠 영역을 구분한다.
- 제목 바로 아래에서 모든 콘텐츠를 문서처럼 순차적으로 쌓지 않는다.
- 콘텐츠 양이 적을 경우 제목을 제외한 본문 영역의 세로 중심에 콘텐츠 그룹을 배치한다.
- 개별 객체가 아니라 콘텐츠 그룹 전체의 외곽 경계를 기준으로 정렬한다.
- 슬라이드 하단에 과도한 빈 공간이 남지 않도록 한다.
- 표, 카드, 차트 및 다이어그램은 중앙 안전 영역 안에 배치한다.
- 배경 장식, 로고 또는 페이지 번호를 핵심 콘텐츠의 정렬 기준으로 사용하지 않는다.
- 레이아웃 참고자료에서는 요소 위치, 간격, 밀도 및 시각적 균형을 확인한다.
- 참고자료의 문구를 실제 내용 원본으로 사용하지 않는다.
- 표지와 Closing은 Urban Navy를 사용할 수 있으며, 정보량이 많은 본문은 Light Theme를 우선한다.
- 표, 카드, 다이어그램 및 차트는 수정 가능한 형태로 구성한다.

PPT 초안을 Markdown으로 작성할 때는 아래 형식을 사용할 수 있다.

```markdown
# 슬라이드 1. 제목

## 핵심 메시지
- ...

## 화면 구성
- 배경:
- 주요 요소:
- 사용할 로고:
- 컬러 테마:
- 적용할 문서 유형 가이드:
- 확인할 레이아웃 참고자료:

## 본문 내용
- ...
```

### 9.4 Word / General Document

Word 작성 시 `document-type-guides`의 Word 제작 가이드를 우선 적용하고, 필요한 경우 `document-templates`를 작업 원본으로 사용한다.

- 기본 용지는 A4를 기준으로 한다.
- 표지, 개정 이력, 목차, 본문 및 부록의 흐름을 문서 목적에 맞게 구성한다.
- 불필요한 페이지 나누기, 구역 나누기 및 빈 문단을 만들지 않는다.
- 표가 페이지 폭을 넘어가지 않도록 한다.
- 행이 페이지 사이에서 읽기 어렵게 분리되지 않도록 한다.
- 여러 페이지 표는 머리글 행을 반복한다.
- 표 뒤의 잔여 문단과 구역 설정으로 빈 페이지가 생성되지 않는지 확인한다.
- 제목, 본문, 표 및 캡션은 Word 스타일과 자동 기능을 우선 사용한다.
- 머리말과 바닥글은 본문을 침범하지 않도록 구성한다.
- 텍스트와 표는 편집 가능한 형태로 유지한다.

### 9.5 Excel

- 데이터 입력, 정렬, 필터 및 열람 편의성을 장식보다 우선한다.
- 데이터 영역 전체에 어두운 배경을 사용하지 않는다.
- Urban Navy는 제목, 구분 영역 또는 표 헤더에 제한적으로 사용한다.
- 표지는 문서 목적에 따라 선택적으로 사용한다.
- 수치와 날짜 형식을 일관되게 유지한다.
- 인쇄가 필요한 경우 용지 방향, 배율, 인쇄 영역 및 페이지 나누기를 검토한다.
- 데이터와 수식을 이미지로 변환하지 않는다.

---

## 10. Table Rules

- 표 헤더는 가운데 정렬한다.
- 긴 문장, 설명 및 비고는 왼쪽 정렬한다.
- 상태, 구분, 여부 및 짧은 코드값은 가운데 정렬한다.
- 비교 목적의 수치는 오른쪽 정렬한다.
- 단위, 기준일 및 조건은 표 주변에 명확히 표시한다.
- 표 전체를 일괄 가운데 정렬하지 않는다.
- 강한 색상으로 셀 전체를 과도하게 채우지 않는다.
- 상태 의미에는 Semantic Color와 텍스트 라벨을 함께 사용한다.
- 복잡한 표는 요약 표와 상세 표로 분리한다.

---

## 11. Long Document Rules

긴 문서는 아래 순서로 작성한다.

1. 문서 구조 초안
2. 섹션별 본문 작성
3. 표, 카드 및 다이어그램 정리
4. 디자인 기준 적용
5. 최종 검토

- 긴 문서를 한 화면이나 한 페이지에 압축하지 않는다.
- 정보량이 많으면 섹션, 페이지 또는 슬라이드로 분리한다.
- 글자 크기와 여백을 과도하게 줄여 억지로 맞추지 않는다.

## 12. Do Not

- Vatos Cyan을 성공 또는 개선 의미로 사용하지 않는다.
- Supporting Colors를 브랜드 메인 컬러처럼 사용하지 않는다.
- Premium Dark를 검은색 메인 배경처럼 남용하지 않는다.
- 순수 Black `#000000`을 기본 텍스트로 사용하지 않는다.
- 로고를 임의로 변형하지 않는다.
- 원문에 없는 내용을 임의로 추가하지 않는다.
- 고객 제출용 문서를 가볍거나 장난스럽게 표현하지 않는다.
- AI 또는 노션 특유의 캐주얼한 카드형 디자인을 그대로 사용하지 않는다.
- 승인되지 않은 임의 스타일을 새로 만들지 않는다.
- 외부 CDN에 의존하는 문서를 만들지 않는다.
- 통이미지 형태의 문서를 만들지 않는다.
- 과도하게 넓은 Markdown 표를 만들지 않는다.
- 특정 파일명이나 경로에만 의존하는 지침을 작성하지 않는다.
